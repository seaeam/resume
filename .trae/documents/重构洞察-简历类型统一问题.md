# 1. 问题

当前代码在“简历类型”的枚举值与默认值上存在不一致：不同模块分别使用了“default”和“basic”两套含义接近但不相同的标识。

这会导致在不同流程中出现类型值不一致、状态回写混乱、以及后续扩展（新增样式类型）时的歧义。

## 1.1. **枚举值不一致：default vs basic**
- 位置：
  - src/store/resume/form.ts：`type: 'basic' | 'modern' | 'simple'`（默认 `'basic'`）
  - src/store/resume/current.ts：`ResumeType = 'default' | 'simple' | 'modern'`
  - src/lib/offline-resume-manager.ts：创建离线简历默认 `type: 'default'`
  - src/lib/supabase/resume/form.ts：创建云端简历默认 `type: 'default'`
- 为什么是问题：
  - 在加载/保存链路中，类型值可能在不同层面产生不一致（如 UI 用 `'basic'`，元信息存 `'default'`），导致样式选择和配置不同步。
  - 后续新增类型时，不一致的枚举基线会让判断分支与迁移逻辑变复杂，增加理解与维护成本。

示例代码（问题现状）：
```ts
// src/store/resume/form.ts
interface ResumeState {
  type: 'basic' | 'modern' | 'simple'
}

// src/store/resume/current.ts
export type ResumeType = 'default' | 'simple' | 'modern'

// src/lib/offline-resume-manager.ts
const resume = {
  // ...
  type: options.type || 'default',
}

// src/lib/supabase/resume/form.ts
export async function createNewResume(info, type: string = 'default') { /* ... */ }
```

## 1.2. **默认值不一致导致的状态回写歧义**
- 位置：src/lib/automerge/document-manager.ts `mapDocToState` 与 `changeType` 的协同。
- 为什么是问题：
  - `mapDocToState` 将 `type` 直接取 `source?.type || 'basic'`，若数据库为 `'default'`，运行时就会出现 `'default'` 与 `'basic'` 混用；随后 `changeType` 又把 `doc.type` 写回为 `'basic' | 'modern' | 'simple'`，使文档与配置在不同阶段出现值不一致。

示例代码（问题现状）：
```ts
// mapDocToState
return {
  // ...
  type: source?.type || 'basic',
}

// changeType
applyResumeChange(set, get, { type }, (doc) => { doc.type = type })
```

# 2. 收益

统一“简历类型”的枚举与默认值，确保 UI 状态、离线存储与云端配置在整个链路中一致。

## 2.1. **降低复杂度与歧义**
- 消除 `'default'` 与 `'basic'` 的混用后，判断分支更少、代码更易懂。
- 估计能使与类型相关的分支减少到 **1 套**统一逻辑。

## 2.2. **提高一致性与可测试性**
- 统一后的枚举可在单测中集中校验，不再需要针对不同来源写特殊分支。
- 端到端测试只需验证一种类型基线，回归成本降低。

## 2.3. **增强可扩展性**
- 未来新增类型（如 `'compact'`）时，只需在统一枚举与映射表中补充，无需改动多处默认值。

# 3. 方案

整体思路：建立统一的 `ResumeType` 定义与“旧值到新值”的规范化映射；在创建、加载、保存的关键路径上应用该映射，完成一次性收敛。

## 3.1. **统一类型定义与规范化工具**

新增一个集中管理的工具模块（例如 `src/lib/resume-type.ts`）：
```ts
// src/lib/resume-type.ts
export type ResumeType = 'basic' | 'modern' | 'simple'

const LEGACY_TO_NEW: Record<string, ResumeType> = {
  default: 'basic',
  basic: 'basic',
  modern: 'modern',
  simple: 'simple',
}

export function normalizeResumeType(input?: string | null): ResumeType {
  return LEGACY_TO_NEW[(input || '').toLowerCase()] || 'basic'
}
```

## 3.2. **在创建与离线存储层统一默认值**

- 修改离线创建默认：
```ts
// src/lib/offline-resume-manager.ts
const resume = {
  // ...
  type: options.type ? normalizeResumeType(options.type) : 'basic',
}
```

- 修改云端创建默认：
```ts
// src/lib/supabase/resume/form.ts
export async function createNewResume(info, type: string = 'basic') { /* ... */ }
```

## 3.3. **在加载映射处进行规范化**

- `mapDocToState`：
```ts
// src/store/resume/form.ts（或 document-manager.ts 的映射）
return {
  // ...
  type: normalizeResumeType(source?.type),
}
```

- `changeType` 的类型来源统一为新枚举：
```ts
// src/store/resume/form.ts
changeType: (type: ResumeType) => {
  applyResumeChange(set, get, { type }, (doc) => { doc.type = type })
}
```

- `useCurrentResumeStore` 统一：
```ts
// src/store/resume/current.ts
export type ResumeType = 'basic' | 'simple' | 'modern'
// setCurrentResume(id, 'basic') // 默认使用统一枚举
```

## 3.4. **前后对比**

问题前：
```ts
// 多处默认值与枚举不一致
createNewResume(..., type = 'default')
createOfflineResume(... type = 'default')
ResumeState.type = 'basic' | 'modern' | 'simple'
CurrentResume.ResumeType = 'default' | 'simple' | 'modern'
```

问题后：
```ts
// 统一为一个枚举与默认值
export type ResumeType = 'basic' | 'modern' | 'simple'
createNewResume(..., type = 'basic')
createOfflineResume(... type = 'basic')
mapDocToState => type = normalizeResumeType(source?.type)
CurrentResume.ResumeType = 'basic' | 'simple' | 'modern'
```

# 4. 回归范围

从端到端流程验证“类型一致性”在离线、在线与协作场景下均正确。

## 4.1. 主链路
- 创建离线简历 → 进入编辑器 → 切换样式类型 → 关闭并重新打开页面 → 类型保持为 `'basic' | 'modern' | 'simple'` 的统一枚举。
- 创建云端简历 → 进入编辑器 → 切换样式类型 → 手动/自动保存 → 重新加载 → 类型与 UI 一致。
- 开启协作会话（发起者与加入者）→ 双方查看类型 → 切换样式 → 双方保持一致并可回写。

## 4.2. 边界情况
- 旧数据包含 `'default'`：加载后统一映射为 `'basic'`；切换与保存不再产生 `'default'`。
- 无 `type` 字段或值非法（如 `'unknown'`）：归一化到 `'basic'` 并保持一致。
- 并发编辑下类型切换：确保 Automerge 变更合并后最终仍是统一枚举集合中的合法值。
