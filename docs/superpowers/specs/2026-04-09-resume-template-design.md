# 简历模板系统设计

**日期：** 2026-04-09

**目标：** 将 `/template` 明确定义为“简历模板”页面，而不是“模板工作台”；同时建立三类模板库、模板直接创建简历的使用链路、用户自定义模板链路，以及可用的模板编辑器体验。

**替代说明：** 本文档替代 `docs/superpowers/specs/2026-04-08-resume-template-editor-design.md` 中对第二阶段产品形态的定义。第一阶段运行时底座设计 `docs/superpowers/specs/2026-04-08-resume-template-runtime-design.md` 继续有效。

---

## 当前偏差

当前实现与目标之间有四个核心偏差：

- `/template` 目前更像“模板工作台”，而不是用户能浏览和挑选简历模板的模板库。
- 模板分类只有“官方模板 / 我的模板”，缺少“社区模板”这一层产品定义。
- “直接使用模板创建简历”还没有成为系统级链路，当前底座仍主要围绕 `ResumeType` 和 built-in manifest 运作。
- 模板编辑器本身仍不可用：
  - 实时预览没有走自适应缩放，A4 画布在桌面和移动端都可能溢出。
  - 预览内容只覆盖了少量 section renderer，不能展示完整简历字段。
  - 三栏布局在窄屏下没有收敛为可用的响应式结构。
  - 结构面板拖拽排序后顺序没有正确写回 manifest。

这意味着第二阶段不能继续按“模板工作台”补功能，而必须先纠正产品定义，再修编辑器的关键缺陷。

## 正确的产品定义

### 1. `/template` 页面名称与职责

`/template` 页面名称统一为 **简历模板**。

它的职责是：

- 浏览官方模板
- 浏览社区模板
- 管理我的模板
- 从模板直接创建简历
- 从模板复制后进入自定义模板编辑

它不是模板编辑器首页。模板编辑器只是“简历模板”页面中的二级流程。

### 2. 三类模板

系统内存在三类模板：

- `官方模板`
  - 平台提供的内置模板。
  - 这些模板主要通过 `TemplateManifest` 的布局、颜色、字体、间距、头部样式和默认 section 编排来区分。
- `社区模板`
  - 用户发布出去的模板。
  - 非官方，所有权属于原作者。
  - 其他用户可以使用或复制，但不能直接编辑原模板。
- `我的模板`
  - 当前用户拥有的模板。
  - 包括从官方模板复制、自定义后保存的模板，以及从社区模板复制过来的模板。
  - 不管私有还是已发布，都应在这里统一管理。

### 3. 模板不是“另一种简历内容”

简历模板表达的是以下差异：

- 布局骨架，例如单栏、左侧栏、右侧栏、分段式
- 头部样式
- 颜色预设
- 字体预设
- 间距预设
- 圆角与视觉风格
- section 的顺序、显隐、主栏/侧栏归属

模板不表达业务数据本身。简历内容仍然属于 `ResumeSchema`。

## 设计原则

### 1. 模板库优先，编辑器次级

用户访问 `/template` 的第一目标应该是“选模板”，不是“进工作台”。

因此默认视图必须是模板库。进入编辑器只发生在：

- 从官方模板点击 `自定义模板`
- 从社区模板点击 `自定义模板`
- 从我的模板点击 `继续编辑`

### 2. 官方模板和社区模板都不可直接改原件

官方模板和社区模板都属于“来源模板”。

它们只能：

- 被直接用于创建一份新简历
- 被复制成当前用户的私有模板后再编辑

当前用户只能直接编辑“我的模板”。

### 3. “直接使用”创建的是简历实例，不是模板副本

模板卡上的 `直接使用` 动作，不应该创建一份新的模板，而应该：

1. 基于所选模板创建一份新的简历实例
2. 把这份简历与模板建立绑定
3. 跳转到简历编辑页继续编辑内容

### 4. 编辑器必须复用已有可自适应的预览能力

模板编辑器中栏不允许继续手写固定 `210mm` 画布容器。

应复用仓库里已经存在的可自适应缩放预览实现，至少沿用 `src/components/resume/scaled-readonly-preview.tsx` 的测量与缩放思路，抽成共享预览容器或直接复用已有组件逻辑，使预览在桌面与移动端都能完整可见。

### 5. 拖拽排序必须以 region 内顺序为准

模板 section 的排序应表达为“主栏内部顺序”和“侧栏内部顺序”的结果，而不是被全局 `order` 重新洗掉。

同栏拖拽和跨栏拖拽都必须能稳定写回 manifest。

## 页面结构

### 模板库视图

`/template` 默认进入模板库视图。

顶部结构：

- 页面标题：`简历模板`
- 简短说明：突出“选择模板创建简历，或复制后自定义”
- `Tabs`
  - `官方模板`
  - `社区模板`
  - `我的模板`

### 模板卡信息

每张模板卡至少展示：

- 缩略预览
- 模板名称
- 简短描述
- 布局标签，例如：
  - `单栏`
  - `双栏`
  - `侧栏`
- 风格标签，例如：
  - `正式`
  - `现代`
  - `紧凑`

不同 tab 的附加信息：

- `社区模板`
  - 作者信息
  - 发布时间或更新时间
- `我的模板`
  - 当前状态：`私有` / `已发布`

### 模板卡动作

#### 官方模板

- `直接使用`
- `自定义模板`

#### 社区模板

- `直接使用`
- `自定义模板`

#### 我的模板

- `直接使用`
- `继续编辑`
- `发布` / `取消发布`

## 模板来源与列表规则

### 官方模板

官方模板继续来自本地 registry，但不能只停留在两张模板。

首批官方模板应覆盖多种布局与视觉差异，至少包括：

- 经典单栏
- 现代左侧栏
- 现代右侧栏
- 紧凑 ATS
- 强调技能信息的双栏模板
- 分段式现代模板

这些模板本质上仍是 built-in `TemplateManifest`，只是在 `officialTemplateCatalog` 中补充更多清晰可选的 manifest 实例。

### 社区模板

社区模板来自“其他用户已发布的模板”。

第一版列表规则：

- 数据来源：`resume_templates` 中 `visibility = published`
- 默认排除当前用户自己的模板，避免与“我的模板”重复管理
- 只展示用户模板，不混入官方模板

### 我的模板

“我的模板”展示当前用户自己的全部模板：

- 私有模板
- 已发布模板

这部分是用户的模板管理入口。

## 关键用户流

### 1. 直接使用官方模板

1. 用户进入 `简历模板`
2. 在 `官方模板` 中选择某张模板
3. 点击 `直接使用`
4. 系统创建一份新的简历实例，并绑定该模板
5. 跳转到简历编辑页

### 2. 直接使用社区模板

1. 用户进入 `社区模板`
2. 选择某张社区模板
3. 点击 `直接使用`
4. 系统创建一份新的简历实例，并绑定该模板
5. 跳转到简历编辑页

### 3. 基于官方模板自定义

1. 用户在 `官方模板` 点击 `自定义模板`
2. 系统先复制一份私有模板到“我的模板”
3. 再进入模板编辑器
4. 用户保存后，该模板继续归入“我的模板”

### 4. 基于社区模板自定义

1. 用户在 `社区模板` 点击 `自定义模板`
2. 系统复制一份私有模板到“我的模板”
3. 原社区模板不被修改
4. 用户进入模板编辑器编辑自己的副本

### 5. 编辑我的模板

1. 用户进入 `我的模板`
2. 点击 `继续编辑`
3. 编辑器加载这份用户模板的 manifest
4. 用户修改后保存

### 6. 发布与取消发布

1. 用户在“我的模板”中点击 `发布`
2. 系统做发布校验
3. 通过后切换为 `published`
4. 模板进入社区模板集合

反向操作：

1. 用户点击 `取消发布`
2. 模板从社区模板集合中移除
3. 模板仍然保留在“我的模板”

## 简历实例与模板绑定模型

当前系统中的简历仍主要依赖 `ResumeType`（`default` / `simple` / `modern`）选择模板。

为了支持“从官方模板 / 社区模板 / 我的模板直接创建简历”，需要在简历实例层新增“模板绑定”概念。

### 设计要求

- 现有 `ResumeType` 继续保留，用于兼容旧简历
- 新创建的简历可以额外携带模板绑定信息
- 预览、编辑器和导出在渲染时优先使用模板绑定，若不存在再回退到旧的 `ResumeType`

### 建议模型

在简历持久化快照中新增可选字段：

```ts
templateBinding?: {
  source: 'official' | 'community' | 'user'
  templateId: string
  basedOnResumeType?: ResumeType
}
```

解释：

- `templateId`
  - 指向官方模板 catalog item 或用户模板记录
- `source`
  - 表示模板来源
- `basedOnResumeType`
  - 作为兼容信息，保留这份简历最初来自哪类 built-in 模板

渲染链路改为：

1. 若存在 `templateBinding`
   - 先解析绑定模板
2. 若不存在
   - 继续走 `getBuiltInTemplateManifest(type)`

这样可以保证：

- 旧简历不受影响
- 新模板库可以正式驱动简历创建

## 编辑器修复设计

### 1. 预览缩放修复

当前中栏预览直接渲染 `PagedResumeShell`，外层只是固定高度滚动区，导致：

- 桌面视图中 A4 画布可能溢出
- 移动端中整页被裁掉
- 用户无法完整观察排版结果

修复方案：

- 抽取或复用 `scaled-readonly-preview.tsx` 中已有的缩放测量逻辑
- 模板编辑器中栏使用“可缩放简历预览容器”
- 预览区域只负责适配 viewport，不再自己决定固定纸张显示尺寸

目标：

- 桌面端完整看到一页缩放简历
- 移动端完整看到缩放后的简历，不再横向溢出

### 2. 完整字段预览

当前 runtime registry 只覆盖了部分 renderer：

- `basics`
- `job_intent`
- `education`
- `work_experience`
- `project_experience`
- `skills`
- `self_evaluation`

这会导致模板编辑器无法展示完整简历信息。

修复要求：

- 模板编辑器默认预览数据必须覆盖完整示例简历
- runtime registry 必须补齐本阶段模板会使用到的 section renderer，至少覆盖现有 schema 中会被模板展示的主要模块
- 如果某些 section 暂不具备专门 renderer，也必须有稳定的降级展示策略，而不是在预览中直接消失

### 3. 响应式布局修复

当前编辑器三栏布局在移动端继续横向并排，导致：

- 面板超出宽度
- 预览区域被挤压裁剪
- 主要内容不可读

修复方案：

- 桌面端保持三栏：
  - 左：结构面板
  - 中：实时预览
  - 右：模板属性
- 窄屏下改为纵向堆叠
- 移动端优先顺序：
  - 顶部工具条
  - 实时预览
  - 结构面板
  - 模板属性

这样移动端进入编辑器时，至少先能看到预览和核心操作。

### 4. 拖拽排序修复

当前问题不是“拖拽组件没动”，而是 manifest 写回逻辑不正确。

修复要求：

- 同栏拖拽后，顺序必须立即体现在 `manifest.sections`
- 跨栏拖拽后，section 的 `region` 和目标栏内顺序都必须更新
- 排序归一化只能维护最终展示顺序，不能覆盖用户刚刚完成的拖拽结果
- 预览应在拖拽完成后立即跟随变化

## 路由与状态策略

为减少本轮改动面，第二阶段仍可维持单路由方案：

- `/template`

通过搜索参数控制状态：

- `tab=official | community | mine`
- `mode=library | editor`
- `source=official | community | user`
- `templateId=<id>`

这样可以做到：

- 默认访问是模板库
- 从任何 tab 进入编辑器后都能返回原 tab
- 不必立刻拆出新的动态路由页面

---

## 需要保留的边界

以下边界继续保持：

- 模板编辑器只读写 `TemplateManifest`
- 简历内容编辑仍归简历编辑页负责
- 模板渲染仍统一走 `ResumeTemplateRuntime`
- 官方模板和社区模板都不能被直接改原件
- 不做自由画布式排版

---

## 完成标准

当以下条件全部满足时，本轮纠偏才算完成：

- `/template` 页面标题和职责已经明确为“简历模板”
- 顶部存在 `官方模板 / 社区模板 / 我的模板` 三个 tabs
- 官方模板和社区模板卡片都提供 `直接使用` 与 `自定义模板`
- 我的模板卡片提供 `直接使用`、`继续编辑`、`发布/取消发布`
- 官方模板数量不再只有两张，而是具备“模板库”感知的多模板选择
- 新建简历可以绑定所选模板，而不仅仅依赖旧的 `ResumeType`
- 模板编辑器中栏预览完整自适应，不再裁剪溢出
- 模板编辑器预览能展示完整简历结构，不再只看到基本信息
- 模板编辑器在移动端不再横向溢出
- 结构面板拖拽排序和跨栏移动都能正确生效
